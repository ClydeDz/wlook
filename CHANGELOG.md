## [0.3.0]

### Added

- **Dashboard UI now follows the Theme dropdown.** Previously the popup and the dashboard shared the same `theme` config field but only the popup applied it (the dashboard surfaced whatever the OS preference was, regardless of user choice). Selecting **Dark** in the Settings panel now turns the dashboard surface — background, cards, section dividers, controls, headings, status dots — dark regardless of OS preference; **Light** forces light; **System (Auto)** follows the OS as before. The change takes effect the moment the dropdown value is saved (no reload required). The dashboard's CSS cascade was refactored to mirror the popup's three-state pattern: `:root` provides the base light tokens, `:root[data-theme="dark"]` overrides for explicit dark, and `@media (prefers-color-scheme: dark) :root:not([data-theme="light"]):not([data-theme="dark"])` covers the system-follow case.

### Changed

- **"Dictionary sources" row in the dashboard About section is now a single inline link** pointing at `https://kaikki.org/dictionary/rawdata.html`, matching the layout of the **"Source"** row directly above it. Previously this row listed two upstream sources — Wiktionary via Kaikki and Open English WordNet — each with their CC BY licence attribution appended inline. The WordNet entry is removed from the dashboard surface entirely; the link text is the URL itself so it visibly reads as a clickable source page rather than a human-friendly label. The attribution chain itself is unchanged: released `.wlpack` files are still under CC BY-SA 4.0, every pack's `metadata.attribution` row still records the upstream licences (see `tools/build-dictionary/src/index.ts:197`), and `docs/SPEC.md` §6.3 still describes the legal reasoning. The "Dictionary packs → Released under CC BY-SA 4.0" row two positions below remains as the user-facing licence summary, so removing the inline attribution here is a deduplication rather than a licence change. The now-unused `.about-licenses` and `.about-licenses li` rules were removed from `src/dashboard/dashboard.css`.
- **Popup theme dropdown now offers three explicit choices: System (Auto), Light, Dark.** Previously the dropdown had a confusing pair: "Default (system)" (which followed the OS) and "Dark" (which forced dark). The new triad mirrors macOS / iOS / Windows settings conventions and surfaces an opt-in Light option that was previously impossible — somebody whose Windows is set to dark but who wanted a light pop-up had no way to get one. The internal config value is now `'system' | 'light' | 'dark'`; `DEFAULT_CONFIG.theme` changed from `'default'` to `'system'`. The dropdown's default order is "System (Auto)" first (matching `DEFAULT_CONFIG.theme`), then Light, then Dark.
- **Legacy `theme: "default"` config values are migrated to `"system"` on read.** `readConfig()` in `src/core/config.ts` translates the old alias to the new internal value so existing user config.json files continue working. The popup's `data-theme="default"` CSS hook is unchanged, so users who never open Settings see identical behaviour. The Settings dropdown would otherwise render with no option selected (browsers don't show a `<select>` value that matches no `<option>`) if the migration had been omitted. Migration is read-time only: the on-disk `theme` value stays as `"default"` until the user opens Settings, at which point the next `updateConfig` write overwrites it. A missing `theme` key (a user with an even-older config.json) is normalised the same way — treated as `'system'` rather than falling back to `DEFAULT_CONFIG.theme`, so a future change to `DEFAULT_CONFIG.theme` cannot accidentally re-introduce the legacy `"default"` value.
- **Global hotkey accelerator strings are uniformly written as `CommandOrControl+Shift+D`** (the Electron-canonical cross-platform prefix). A new pure `normaliseHotkey()` helper exported from `src/core/config.ts` tokenises an accelerator, treats `Ctrl`/`Control`/`Cmd`/`Command` (case-insensitive) as aliases for the platform's primary modifier and rewrites them to a single `CommandOrControl` token, title-cases single-character action keys and the four known multi-character modifiers, and deduplicates repeats while preserving modifiers like `Meta`/`Alt`/`Shift` verbatim. `mergeConfig()` invokes it on `raw.hotkey` so legacy config.json values (typically `"Ctrl+Shift+D"` from hand-edits or the pre-canonical recorder output) are silently migrated to `CommandOrControl+Shift+D` on next read. `src/dashboard/components/Settings.tsx` recorder and its "Reset to default" button emit the canonical form directly; `src/dashboard/components/SystemHealth.tsx` display fallback is updated to match. Active binding behaviour is unchanged on Windows — both `Ctrl+Shift+D` and `CommandOrControl+Shift+D` resolve to the literal Ctrl binding — but on-disk values are now stable, grep-friendly, and portable to macOS if Wlook ever ships there. A `"hotkey": null` config entry is correctly preserved as the legacy disable marker documented at `docs/customisation.md` §15.4 — an earlier draft of this rewrite's `typeof !== 'string' ? DEFAULT_CONFIG.hotkey` fallback would have silently re-bound a null value to the canonical default, undoing the user's disable intent. The polish round added an `raw.hotkey === null` short-circuit in `mergeConfig`, so null now passes through verbatim. The recommended way to disable going forward remains `hotkeyEnabled: false`. `WlookConfig.hotkey` is now `string | null` accordingly; `src/agent/ipc.ts`' `get-status` handler coerces a null value to `''` at the IPC boundary so `DashboardStatus.hotkeyAccelerator` stays a non-nullable string and existing consumers keep working unchanged.

### Fixed

- **System Health "Selection capture" row no longer contradicts itself when clipboard fallback is enabled.** The greyed-out note on the clipboard-fallback row used to read "Enable in Settings for apps without UIA support", which fired whenever the agent's selection-capture method was clipboard fallback — i.e., exactly the state where the user has already enabled the setting in **Dashboard → Settings**. The buggy line in `captureNote()` (`src/dashboard/components/SystemHealth.tsx`) is removed; the label "Selection capture: Clipboard fallback" and the amber dot already convey the active mode, and the `unavailable` note ("Could not read text selection") still fires when neither path succeeds.

### Internal

- `src/popup/popup.ts` `loadConfig()` now switches on the theme value instead of a ternary: `'light'` → `data-theme="light"`, `'dark'` → `data-theme="dark"`, `'system'` (and any unknown/legacy value) → `data-theme="default"` (follow OS). The `WlookAPI.getConfig` return type narrowed from `'default' | 'dark'` to `'light' | 'dark' | 'system'`.
- `src/popup/popup.css` comment block updated to describe the three `data-theme` values. No new CSS rule was needed: the existing `:root:not([data-theme="light"])` inside the `@media (prefers-color-scheme: dark)` block already prevents the dark tokens from applying when the user chooses Light on a dark OS, so the base `:root` light tokens apply naturally.
- `tests/unit/core/config.test.ts` gained a `theme migration` describe block with two cases: legacy `"default"` → `"system"`, and `light`/`dark` left untouched.
- `src/dashboard/dashboard.css`: dark tokens split out of the OS-follow media query into a dedicated `:root[data-theme="dark"]` block. The OS-follow block now asserts `:not([data-theme="light"]):not([data-theme="dark"])` so it and the explicit-dark block are mutually exclusive (no source-order surprise). Five hardcoded `rgba(...)` values — `.btn--danger` border, `.btn--danger:hover` background, `.btn--ghost:hover` background, `.pack-row__preferred-badge` background, and `.health-dot--red` box-shadow halo — replaced with `color-mix(in srgb, var(--token) X%, transparent)` so the tinted surfaces and the danger-status halo follow `--accent` / `--danger` / `--dot-red` instead of light-mode-only RGB.
- `src/dashboard/app.tsx`: new `applyTheme(theme)` helper maps `'light' | 'dark' | 'system'` to the corresponding `<html data-theme="…">` attribute (mirrors `src/popup/popup.ts` exactly). Called from a `useEffect` keyed on `config.theme` so changes from the Settings dropdown take effect immediately without a reload. A `Theme` type alias was lifted above `AppConfig` so the helper, the prop, and the config field reference one name.
- `tests/unit/core/config.test.ts` gained two new top-level describe blocks: `normaliseHotkey` (ten cases covering empty input, the four primary-modifier aliases, case-insensitivity on the modifier token, whitespace trimming, deduplication of repeated tokens, the `Meta` modifier being preserved, single-character action-key upper-casing, and multi-character action keys like `End` being passed through), and `mergeConfig — hotkey migration` (three cases confirming the read-time rewrite actually runs through `mergeConfig`, the canonical form is idempotent, and anon-string raw value falls back to `DEFAULT_CONFIG.hotkey`).
- `DashboardStatus.selectionCaptureMethod` now reflects the **runtime** capture path the agent actually used, not the configured mode (`config.clipboardFallback`). The change spans nine files: (1) `src/agent/selection-capture.ts` `SelectionCapture` holds a persistent instance across captures so its `lastMethod` cache (`'uia' | 'clipboard' | 'unavailable' | null`) survives between hotkey triggers; `captureWindows()` and `captureClipboardDirect()` flip `lastMethod` after every branch and a clipboard fallback that ran but returned `null` now records `'unavailable'` (previously indistinguishable from a successful fallback at the status layer). The module-level `captureCurrentSelection()` one-shot helper is kept for ad-hoc tests but is now `@deprecated` because silently instantiating a fresh `SelectionCapture` would bypass the runtime tracking the rest of this fix establishes. (2) `src/agent/main.ts` owns the persistent instance, calls `selectionCapture.updateConfig(updated)` whenever the renderer pushes a config patch, and routes the hotkey handler through `selectionCapture.capture()`. After every capture (including the no-selection bail) it broadcasts a fresh `DashboardStatus` so the dashboard UIs update in real time. (3) `src/agent/ipc.ts` gains two top-level exports — `buildDashboardStatus(ctx)` (single source of truth for the status payload, used by both `get-status` and the broadcast path) and `pushStatusToRenderers(status)` (fans out to every open `BrowserWindow` via `webContents.send('status-update', status)`). The `IPCContext` interface gains `broadcastStatus: () => Promise<void>` as a closure surface; the `get-status` handler delegates to `buildDashboardStatus(ctx)`. (4) `src/shared/ipc-contracts.ts` widens the field type to `'uia' | 'clipboard' | 'unavailable' | 'unknown'` and adds the `'status-update'` channel; the `unknown` literal collapses the prior platform-specific optimistic default so a fresh tray app, before any capture, now reads as `'unknown'` (neutral grey dot) instead of falsely reporting `'uia'` on Windows. (5) The renderer path: `src/agent/preload.ts` registers a single `'status-update'` listener at module load and routes payloads to a `Set<callback>` for fan-out; the `window.wlook` API exposes `onStatusUpdate(cb)` / `offStatusUpdate(cb)`. `src/dashboard/app.tsx` subscribes once on mount (cleanup on unmount) and replaces its `useState<DashboardStatus>` with each push, so the System Health row flips in real time. `src/dashboard/components/SystemHealth.tsx` widens `DotColor` to include `'grey'` and adds a `'unknown'` branch in `captureColor` / `captureLabel` / `captureNote`. `src/dashboard/dashboard.css` adds a `--dot-grey` token to the light, dark, and system-follow theme blocks plus a matching `.health-dot--grey` selector. Practical user-facing effects: a `clipboardFallback: true` user sees green after a UIA capture (previously amber on that path); amber appears only when the agent just used the clipboard fallback successfully; red fires whenever both paths failed; and **grey with a "Press the global hotkey to test" hint** replaces the prior "no-op green / UI Automation" pre-capture state.

## [0.2.0]

### Added

- Top-level `productName: Wlook` in `package.json`. The packaging-time Product Name was already declared in `electron-builder.yml`; the new package.json field is belt-and-braces so `app.getName()` and electron-builder both resolve to "Wlook" cleanly.
- Popup header logo: the logo SVG (`assets/icons/popup-logo.svg`) is now embedded inline in `src/popup/popup.ts` via a `POPUP_LOGO_SVG` literal, rendered in both `entryTemplate` and `emptyTemplate`. Inlining (rather than `<img src=>`) lets the strokes pick up CSS `currentColor` from the host page, so the logo can be theme-tinted instead of rendering as fixed black.
- Dashboard window taskbar icon: `src/agent/dashboard-window.ts` now loads `assets/icons/app.ico` via `nativeImage.createFromPath` with an `existsSync` fallback and passes it to `BrowserWindow({ icon: ... })`. The popup window remains `skipTaskbar: true` by design.
- `--logo-color` CSS variable in `src/popup/popup.css`. Defaults to `var(--accent)` in light mode; `:root[data-theme="dark"]` and the `@media (prefers-color-scheme: dark)` fallback override it to `#ffffff` so the popup logo renders white on dark surfaces.
- `--link-color` CSS variable for the popup's "Search on Google" footer link. Defaults to `var(--accent)` in light mode; both dark-mode selectors override it to `#ffffff`.
- Hover/focus-visible state for `.card__google-link` now draws a 1 px dotted underline 3 px below the text, in `var(--link-color)` (so white on dark, blue on light).
- `app.setName('Wlook')` call in `src/agent/main.ts`, placed before `app.setAppUserModelId('com.wlook.app')`. Combined with the new `productName` in `package.json`, this ensures "Wlook" appears in userData folder naming, login-item metadata, and any Electron code path that derives a display name from `app.getName()`.
- `LICENSE` file at the repo root containing the MIT license text, copyright Clyde D'Souza, 2026. The codebase had no prior license declaration, so by default no one had explicit permission to copy or modify it; adding MIT strictly improves the situation.
- `"license": "MIT"` field in `package.json` so package managers, GitHub's repo header detection, and downstream tooling can identify the license without parsing the LICENSE file.

### Changed

- `DEFAULT_CONFIG.catalogueUrl` now points at the upstream (`clydedz/wlook`) GitHub Releases manifest instead of `null`, so the "Browse Dictionaries" tab in the dashboard works on first run. Users can override per-install in `%APPDATA%\Wlook\config.json`; set to `null` to fall back to the local-drop empty state.
- `ManifestPack.sha256` is now optional. Packs declared in `packs-manifest.json` without a `sha256` field install without integrity checking. Mismatched hashes still fail the install. **Trade-off:** this weakens the supply-chain check that previously protected against a tampered `.wlpack` from a compromised manifest host or hostile CDN edge. Publishers who care about integrity should keep shipping `sha256`.
- `src/popup/popup.ts` `loadConfig()` now reads `config.theme` from the `get-config` IPC response (previously dropped on the floor; only `popupSearch` was consumed). It sets `document.documentElement.setAttribute('data-theme', theme === 'dark' ? 'dark' : 'default')`. The `WlookAPI.getConfig` return type was extended accordingly.
- `src/popup/popup.css` theme switching refactored away from a pure OS-preference `@media (prefers-color-scheme: dark)` rule. The dark-surface colour block now also fires when `:root[data-theme="dark"]`, honouring an explicit user choice regardless of OS theme. The existing `assets/themes/default.css` and `dark.css` files remain in-tree as documentation but the runtime now drives theme via the new popup-side selectors.
- Dashboard render order in `src/dashboard/app.tsx`: `<SystemHealth>` was moved from position #2 (right after `<Header>`) to just above `<About>`. Final order is `Header → InstalledDictionaries → BrowseDictionaries → PreferredDialect → Settings → SystemHealth → About`.
- Dashboard render order: `<PreferredDialect>` was moved from position #2 (right after `<Header>`) to between `<BrowseDictionaries>` and `<Settings>`, so the preferred-dialect decision visually follows "what you can install" rather than appearing before browsing.
- `get-config` IPC handler in `src/agent/ipc.ts` now returns `{ ...ctx.getConfig(), version: app.getVersion() }`. Previously the renderer's `AppConfig.version` field resolved to `undefined` at runtime (TS-permissive contract, runtime-empty), so the dashboard header rendered as `Wlook v` with nothing after it.
- `assets/icons/popup-logo.svg` path strokes changed from hardcoded `#1D1D1F` to `stroke="currentColor"`, allowing the CSS-driven `--logo-color` to re-tint the icon for light and dark themes. `stroke-width`, `stroke-linecap`, and `stroke-linejoin` are unchanged.

### Fixed

- `yarn test` now runs vitest inside Electron's Node runtime (`ELECTRON_RUN_AS_NODE=1`), so the `better-sqlite3` binary built by `electron-rebuild` in `postinstall` loads cleanly under ABI 125. Resolves `ERR_DLOPEN_FAILED` failures in `tests/unit/core/pack-manager.test.ts` and `tests/unit/core/sqlite-provider.test.ts` that occurred when running tests under plain Node 22 (ABI 127). Added `cross-env` for cross-platform env-var propagation on Windows.
- Windows system tray icon was missing because `src/agent/tray.ts` looks up `tray.ico` literally on Windows (`process.platform === 'win32' ? 'tray.ico' : 'tray.png'`), but the icon file was placed at `assets/icons/wlook.ico`. Renamed to `assets/icons/tray.ico`. The `[Tray] Icon not found …` warning no longer fires.
- Windows taskbar right-click context menu and Alt-Tab switcher were showing "Electron" instead of "Wlook" because Windows reads the running executable's PE version-resource Product Name. `app.setName('Wlook')` + the new `productName` in `package.json` cover the Electron-side identity; the EXE itself is rewritten only at packaging time via `electron-builder`'s `productName` field, which was already correct in `electron-builder.yml`. Dev-mode won't show "Wlook" in the taskbar until `yarn package` produces a packaged build, but all in-process app-name resolution now resolves correctly from first launch.
- Dashboard header rendered `Wlook v` with no version string (the trailing whitespace rendered as nothing). Fixed by spreading `app.getVersion()` into the `get-config` IPC response.
- User's theme choice from the Dashboard's Settings panel was silently dropped on the popup: the value was saved to `config.json` but never propagated to the popup template. The popup now reads `config.theme`, applies `data-theme` to `<html>`, and the dark-mode selectors in `popup.css` respond.
- Popup logo rendered nearly black in dark mode even after switching the SVG to `currentColor`. Cause: `<img src="popup-logo.svg">` loads the SVG as an isolated document, so `currentColor` falls back to canvas text colour (black) rather than inheriting from `.card__logo { color: var(--accent) }`. Fixed by inlining the SVG content into the popup template.
- "Search on Google" link stayed Apple-blue in dark mode and provided no hover affordance. Fixed by introducing `--link-color` (white in dark) and a dotted underline on hover/focus that follows the same colour.

## [0.1.0]

### Added

- Initial project scaffold
- Core dictionary engine with SQLite/FTS5 backend
- DictionaryProvider and DictionaryResolver interfaces
- English lemmatizer (irregular verbs, plurals, regular suffixes)
- Cross-dialect lookup (all en-\* packs searched, preferred dialect first)

- Frameless popup with definition, IPA, POS, example
